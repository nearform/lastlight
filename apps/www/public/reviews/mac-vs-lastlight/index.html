The bottom line

A good trade, not yet a finished one

MAC is the better library architecture and a dramatic simplification of the runtime; Last Light is the better finished, production-hardened product and the more operator-maintainable authoring surface.

The port is best read as deleting most of Last Light's bespoke infrastructure in exchange for a maintained platform — a sound trade. But today it does so at the cost of the security envelope, the operational control plane, and the multi-tenant deployment story, and it leaves substantial newer Mastra leverage (Channels, Signals, real Memory) on the table while drifting behind the current release line.

Version correction. MIGRATION.md pins @mastra/core 1.37.1, but MAC's lockfile is actually on 1.41.0; latest published is 1.50.1. There is no 2.x — the entire gap is same-major and additive, so upgrading is low-risk. MAC is ~9 minors behind, not a major version behind.

The core answer

What the Mastra port really gains

The strategic win is code MAC no longer maintains because Mastra provides it. A large fraction of Last Light's bespoke surface is retired for a maintained dependency:

  • 01Chat — SessionManager + messaging_sessions/messaging_messages + JSONL chat shim + ChatSessionReader + bespoke pi-ai ChatRunner collapse into ~7 lines of Mastra Memory. biggest single simplification
  • 02Observability — JSONL envelope shim + SessionReader + stat rollups + custom React/Vite dashboard → free AI tracing + Studio.
  • 03Workflow runtime — YAML runner + resume substrate → native createWorkflow / .suspend()/.resume() + durable LibSQL snapshots.
  • 04Sandbox orchestration — gondolin/docker + egress config generation → Mastra Workspace behind a swappable factory.
  • 05Provider registry / OpenCode runtime → model-router strings; per-run env splice → RequestContext DI.

The EventEnvelope contract survives the port essentially intact — the single strongest piece of evidence that Last Light's core domain model was well-designed and that the port didn't compromise it.

The cost

What MAC gives up — and must weigh

✓ Gained

  • Trivial horizontal scale via pluggable cloud sandboxes (e2b / daytona / modal) — no host KVM, no per-host sidecar topology.
  • Radically simpler ops — one MAC_SANDBOX env var vs generated nginx/coredns/otel configs on two internal networks.
  • Free debugging observability — per-step spans, tool I/O, sensitive-data filtering, replay.
  • Enforced modularity — acyclic package graph asserted in CI.
  • Token downscoping preserved — the one hard GitHub control that transferred verbatim.

✕ Given up

  • The security envelope — default-deny egress allowlist, SNI-peek firewall, SSRF/metadata floor, agent-in-sandbox containment. All gone.
  • The operational control plane — approvals UI + history, resume/cancel/retry, daily/hourly stat rollups, session timeline. The dashboard was deleted, not replaced.
  • Multi-tenant deployment — overlay/instance layering + core-version pinning. No answer beyond forking the repo.
  • Slack durability — at-least-once webhook delivery + restart-durable threads (now an in-memory Set).
  • DAG runner + crash-loop circuit breaker; single-model only.

#1 production blocker. MAC runs the agent + host provider keys + token-bearing read tools on the host; only shell execute_command is sandboxed. A prompt-injected agent has host keys + open network — a one-request exfiltration path. Cloud providers (e2b/daytona) relocate the shell but are default-open egress: they give isolation, not a default-deny allowlist. Mastra will not provide this for free.

Deep dive · Authoring

Is the workflow / prompt SDL more maintainable?

Last Light's declarative YAML DSL is more maintainable — decisively for operators and prompt authors, modestly even for core devs on the authoring loop. Adding a workflow in Last Light is one file-drop that self-wires its own routable intent, with a real template engine ({{#if}}, dotted vars, automatic phase-output injection) so prompt + data iteration is a markdown edit. In MAC it's ~200–1,300 lines of imperative TS across three files plus a rebuild; injecting a phase output means hand-splicing typed state; and there is no overlay path — only a fork of the server app.

Winner by persona

PersonaWinnerWhy
Deployment operatorLast LightAdd/override whole workflows, prompts, skills, per-phase model/egress via overlay — no checkout, no rebuild. MAC operators can override prompt prose only.
Prompt authorLast LightTemplate engine makes prompt+data iteration a markdown edit; MAC forces every data change through TypeScript + a build.
Core developerSplitMAC's compile-time zod wiring is safer to refactor; Last Light's orchestration is actually tested (loops, gates, DAG, resume) so it's safer to change behavior in.

Last Light — biggest liability

  • No compile-time safety on the value plane. Mistyped template keys, output_var collisions, and cross-resume {{phaseOutputs}} reads fail silently or only at runtime — guarded by convention, not the type checker.

MAC — biggest liability

  • Authoring is gated to core devs and untested at the orchestration level. Hundreds of lines of imperative TS per workflow, hand-rolled loop caps, no per-phase model/egress, no run-level test, no overlay path.

Deep dive · Developer experience

Ergonomics & repo structure

MAC is the more modular and, in 12 months, the more maintainable design. The package split with a CI-asserted acyclic graph and a createMacApp() composition preset makes boundaries structural rather than conventional — packages/mac/test/dependency-graph.test.ts makes "don't create a cycle / don't depend on the app" a failing test. Mastra Studio + source-only packages give the fastest inner loop: editing any package is picked up with no build step.

Last Light pays none of the monorepo tax — one tsc, one vitest, published to npm and running in production today via a hardened GHCR/compose pipeline, extended through an operator-facing overlay rather than code. Its lastlight CLI is a real control plane (login/oauth/skills/server-lifecycle/workflow/session/logs/stats/approvals); MAC's is a dev launcher + 3 trigger verbs.

The key tradeoff: modularity & reuse (MAC) vs shipped simplicity & operational maturity (Last Light). MAC is the better library architecture; Last Light is the better finished product right now. MAC's ceremony — transpilePackages, source-only 0.0.0 private packages, capability-registry indirection, and a not-yet-publishable package set (Phase 10 pending) — is the price of that modularity.

Deep dive · Sandbox

Scalability & simplicity vs security

MAC is clearly the simpler and more horizontally-scalable model — and it isn't close. One env var + a one-line cloud factory replaces gondolin's KVM dependency and a five-service sidecar mesh (coredns×2 / nginx×2 / otel) whose configs are regenerated from the allowlist at every boot. But that simplicity is bought almost entirely with security: everything the complexity buys — default-deny egress, SNI enforcement, the SSRF floor, and full-runtime containment — is exactly what MAC dropped.

Focus: Channels — the feature you named

Shipped in @mastra/core@1.22.0, Channels is a unified messaging-connector layer attached to an Agent: it auto-generates a per-platform webhook route, normalizes platform events (DMs, mentions, subscribed-thread replies, tool approvals), streams responses with typing indicators, and persists thread subscriptions / dedup / tool-approvals — precisely the restart-durable state MAC's in-memory active-thread Set currently loses. It maps almost exactly onto mac-slack/connector.ts and mac-github/github-webhook.ts.

Realistic split: Channels is Agent-attached and geared to the conversational reply loop — it fits MAC's chat/Harness lane cleanly. For the deterministic build/pr-review trigger path, Signals is the better-matched primitive. Together they'd retire most of mac-slack/connector.ts, mac-github/github-webhook.ts, the router normalization, and engine/dispatch.ts.

Drift risk. MAC uses the pre-rename @mastra/core/harness, which was renamed Harness→AgentController at 1.46/1.47 — its chat surface will break on upgrade (Heartbeats→Schedules at 1.50 similarly). "Buy" only keeps winning if MAC tracks the line; the longer it sits on 1.41 the more its hand-rolled connector/dispatch/memory code re-diverges from primitives Mastra now ships.

Recommendation

Highest-leverage next moves for MAC
  • 01Restore a default-deny egress + SSRF floor. The hard production blocker. Mastra will not give this for free — it must be layered on whichever sandbox backend is used.
  • 02Budget a routine 1.41 → 1.50 upgrade. Same-major, low-risk; fixes the Harness→AgentController rename before it bites.
  • 03Adopt Signals + Channels to retire the hand-rolled connector/dispatch layer.
  • 04Turn on real Memory features (working memory / semantic recall) — you already pay for the platform; collect the leverage.

Keep in Last Light's column as things MAC has no substitute for yet and would need to consciously rebuild if it becomes the future: the declarative overlay-friendly workflow authoring model, the operator control plane, and the security-first sandbox. Those three are the explicit debt to schedule.
